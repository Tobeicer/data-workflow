# 游艺圈文档导航

本目录是游艺圈数据工作流的文档入口，用于说明现行文档、确认需求与原始需求之间的职责边界和维护方式。

`README.md` 只承担导航和文档治理职责，不定义业务规则、技术方案、任务状态或执行证据。具体内容以对应职责文档为准。

## 1. 文档分层

| 文档或目录 | 定位 | 是否允许更新 | 更新时机 |
|---|---|---|---|
| `游艺圈数据工作流总纲.md` | 唯一现行总纲和实际执行基线，统一描述项目定位、职责分工、数据分层、跨来源技术架构、逻辑数据模型、运行契约、平台对接边界、实施路线、验收和下一执行点 | 允许 | 职责、架构、边界、实施任务、进度或执行决策发生变化时 |
| `游艺圈游戏游艺设备完整分类清单.md` | 分类领域专项权威，维护标准分类、适配机型、设备形态、玩法属性、平台类目映射、平台专用关键词和采集范围 | 允许 | 分类体系、映射、关键词或范围判定规则变化时 |
| `requirements/信息整理.md` | 实际落地过程中新增且已确认需求的唯一持续维护入口 | 允许 | 出现新的明确业务需求，或已确认需求发生变化时 |
| `requirements/游艺圈AI行业平台需求框架.md` | 历史平台需求框架，用于回溯完整产品愿景 | 不更新 | 仅在需要历史产品背景时查阅 |
| `project-split/` | 受保护的原始总需求快照，用于需求参考，不是实际执行基线 | 禁止修改或删除 | 仅在需要追溯原始需求时查阅 |

## 2. 阅读顺序

一般任务按以下顺序读取：

1. 阅读 `游艺圈数据工作流总纲.md`，确认当前职责边界、技术方案、实施状态和下一执行点。
2. 涉及具体来源时，阅读 `data-workflow/adapters/<source>/README.md`，确认该来源的能力、命令、字段、质量门禁、停止条件和运行证据。
3. 涉及分类、平台类目、关键词或采集范围时，阅读 `游艺圈游戏游艺设备完整分类清单.md`。
4. 涉及当前 L3 Excel 兼容格式时，阅读项目根目录的 `游艺圈数据导入字段规范_v2.md`。
5. 需要确认后续新增业务要求时，阅读 `requirements/信息整理.md`。
6. 只有明确需要历史产品背景或原始需求时，才读取 `requirements/游艺圈AI行业平台需求框架.md` 和 `project-split/`。

## 3. 权威与冲突处理

文档内容发生冲突时，按以下顺序处理：

1. 用户最新明确指令。
2. `游艺圈数据工作流总纲.md`。
3. 当前来源的 `data-workflow/adapters/<source>/README.md`，仅限来源专属事项。
4. `游艺圈游戏游艺设备完整分类清单.md`，仅限分类、平台映射、关键词和范围事项。
5. 项目根目录的 `游艺圈数据导入字段规范_v2.md`，仅限当前 L3 Excel 兼容格式。
6. `requirements/` 和 `project-split/` 中的历史需求内容。

`requirements/信息整理.md` 用于记录新增确认需求，但不直接替代执行基线。新增需求涉及职责、数据层、技术架构、数据库边界、平台集成或实施顺序时，应同步转化为 `游艺圈数据工作流总纲.md` 中的现行执行规则。

## 4. 更新位置

| 变化类型 | 唯一维护位置 |
|---|---|
| 新增或变更的明确业务需求 | `requirements/信息整理.md` |
| 项目定位、职责分工、数据层级、跨来源架构、逻辑模型、运行契约、数据库边界和平台对接 | `游艺圈数据工作流总纲.md` |
| 实施任务、依赖、进度、验收证据、里程碑和下一执行点 | `游艺圈数据工作流总纲.md` |
| 来源专属能力、命令、批次入口、字段、质量门禁、停止条件、恢复方式和运行证据 | `data-workflow/adapters/<source>/README.md` |
| 分类树、平台类目映射、平台专用关键词、排除词和范围判断 | `游艺圈游戏游艺设备完整分类清单.md` |
| 当前 L3 Excel 字段和兼容投影 | 项目根目录 `游艺圈数据导入字段规范_v2.md` |

## 5. 维护原则

- `project-split/` 必须保持原貌，不得修改、删除或将其旧同步规则作为当前执行规则。
- `requirements/游艺圈AI行业平台需求框架.md` 作为历史需求参考保留，不继续同步更新。
- 新确认需求只写入 `requirements/信息整理.md`，不直接改写受保护的原始需求和历史框架。
- 根目录现行文档随项目推进同步维护，确保执行规则、任务状态和下一执行点保持有效。
- 同一事实只在最高职责文档中定义，其他文档通过引用建立联系，不重复维护。
- 正式文档只保留当前有效结论，不记录方案比较、聊天推理、命令失败、重试过程或过期下一步。
- 历史需求中的商城、支付、小程序、APP、Web 后台、交易和正式数据库写入等内容，不自动构成当前数据工作范围；是否执行以现行总纲为准。
- 新增、删除或重命名核心文档时同步更新本导航，但不得在本文件复制动态任务状态和具体实施内容。
