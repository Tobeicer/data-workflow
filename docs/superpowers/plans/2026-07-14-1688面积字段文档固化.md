# 1688 面积字段文档固化 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在不启动 1688 全量采集、不修改解析代码和历史批次的前提下，将“工厂面积”和“厂房面积”作为两个独立字段写入所有相关现行文档。

**Architecture:** 执行基线定义跨来源的语义隔离原则，1688 来源指南定义来源标签和标准字段，L3 v2 只定义当前消费映射，1688 设计与实施文档引用同一决定。字段事实只在最高权威文档完整定义，其他文档不扩展口径。

**Tech Stack:** Markdown、PowerShell、ripgrep、Git

## Global Constraints

- `工厂面积` 固定映射为 `factory_area_sqm`。
- `厂房面积` 固定映射为 `factory_building_area_sqm`。
- 两个字段独立保存，不互相回填，不把二者数值差异标记为同字段冲突。
- 1688 页面未解释口径时，只保留原始标签、数值和证据，不自行推断面积组成。
- 当前五商品、四公司批次是试采验证，不是全量采集成果。
- 本计划不启动 1688 全量或扩大采集，不修改 Python、JSON Schema、自动化测试和历史 L0-L2。
- 不修改 `docs/project-split/`、`docs/requirements/` 及其内容。
- 不写入 `public.*` 正式业务表。
- 每次提交只包含当前任务列出的文档，不夹带工作区已有修改。

## Subsequent Execution Roadmap

本文档固化完成后，按以下顺序推进后续独立工作；每一项在执行前单独设计和计划：

1. **解析契约拆分**：先写失败测试，证明信用档案“厂房面积”不能继续写入 `factory_area_sqm`；再修改解析器、`1688_company_asset.schema.json` 和测试，输出两个独立字段。
2. **历史兼容策略**：保留 L0 不变；对现有试采批次从原始证据重新生成新的 L1/L2，旧结果只读归档，不原地修改原始响应。
3. **字段差异发现试采**：覆盖普通工厂、超级工厂和普通店铺等页面形态，建立公司字段并集、页面来源矩阵、缺失原因和解析漂移报告。
4. **扩大采集门禁**：字段新增趋于稳定、页面形态有固定样本、两个面积字段测试通过、登录/验证码/限流状态明确后，才扩大公司队列。
5. **批量公司采集**：以商品为入口、按 `memberId` 去重公司；保存多页面 L0、完整 L1 字段、L2 快照/冲突/质量和统一运行状态。
6. **质量与复核**：逐公司输出字段覆盖率，区分未公开、页面缺失、访问受限、解析失败和真实空值；同字段多值进入复核，不同字段并存不告警。
7. **统一编排**：批次契约稳定后接入 n8n 的人工登录门禁、重试、断点、熔断、告警和运行报告。
8. **平台消费适配**：平台正式契约到达后再决定是否增加“工厂面积”消费列；L3 变化不得删除或合并 L1-L2 字段。
