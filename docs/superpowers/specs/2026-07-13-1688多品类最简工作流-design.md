# 1688 多品类最简工作流设计

日期：2026-07-13  
状态：用户已确认实施  
上位基线：`../../数据工作流与游艺圈系统对接执行基线.md`

## 1. 目标

将现有单商品公司试采收束为一套可批量执行、可去重、可追溯的 1688 最简工作流，并用 5 个商品验证娃娃机、弹珠机和老虎机相关商品的信息获取链路。

验证样本：

- 娃娃机 2 个，优先选择同一公司不同商品，验证公司去重；
- 弹珠机 2 个，优先选择不同公司；
- 老虎机相关商品 1 个，来源名称允许为拉霸机、水果机等，但保留原始标题，不直接完成业务分类；
- 合计尽量覆盖至少 4 家公司。

## 2. 方案选择

采用“最简主流程 + L0 补充证据备案”。

不把所有页面和响应都作为主流程节点，也不另建一套独立备案工作流。主流程只依赖稳定入口和结构化结果；同页发现的其他响应、HTML、文本、截图、图片和暂未消费字段统一随批次进入 L0。

这样可以同时满足：

- 主流程短、状态清楚、适合 n8n 编排；
- 原始信息不丢失，后续新增字段可以离线重放；
- 页面临时缺少工厂或证书信息时，不阻断商品资产保存；
- 同一公司多个商品不会重复访问公司页面。

## 3. 最简主流程

```text
关键词搜索/候选输入
→ 样本选择与商品去重
→ 逐商品采集详情和 SKU
→ 从 sellerModel 提取 memberId/店铺
→ 按 memberId 建立公司任务队列
→ 每家公司采集一次公司档案、联系方式和官方主体资质
→ 输出 L1 商品/店铺/公司/工厂/证书/专利
→ 输出 L2 关系、质量、冲突和复核队列
→ 生成 run_manifest.json 与 run_result.json
```

正式主体资质来源为 1688 `businessinfor.html` 和 `wp_pc_shop_basic_info`。88查、企查查不进入主流程。

## 4. 数据分层

### L0 补充证据备案

- 搜索候选和样本选择结果；
- 商品、店铺、公司档案、联系方式和主体资质原始 HTML/文本；
- 相关 API 原始响应和 URL 清单；
- 调试截图、页面哈希和采集时间；
- 暂未使用的字段、图片和接口错误。

### L1 标准化资产

- `products.jsonl`：商品基础信息、完整属性和公司候选；
- `skus.jsonl`：规格、价格、库存和 SKU 图片；
- `shops.jsonl`：店铺和公开经营信息；
- `companies.jsonl`：工商主体；
- `factory_profiles.jsonl`：带时间和来源口径的工厂快照；
- `certificates.jsonl`、`patents.jsonl`、`contacts.jsonl`、`company_media.jsonl`。

### L2 关系与质量

- 商品—店铺、商品—公司、店铺—公司关系；
- 公司任务去重结果；
- 身份冲突和复核队列；
- 商品链路与公司链路分别统计的质量报告；
- 批次级运行结果。

## 5. 组件边界

### 样本选择器

从搜索候选中按品类数量选择 5 个商品，保留选择原因。选择器是纯函数，不访问线上页面。

### 商品解析器

浏览器只提取页面结构化数据；纯 Python 解析器负责生成商品和 SKU 记录。未知价格、库存或字段使用空值，不填 0。

### 多商品编排器

先完成全部商品页，再按 `memberId` 去重公司任务。公司页面只访问一次，但每个商品都生成自己的关系记录。

### 浏览器适配器

复用现有持久化登录 profile。遇到登录、验证码、入口缺失或结构变化时保存当前证据并返回明确状态，不绕过。

## 6. 状态和错误处理

商品与公司分别记录状态：

- `success`
- `partial_success`
- `login_required`
- `human_verification_required`
- `company_entry_missing`
- `parser_drift`
- `error`

单个商品失败不删除已完成商品；公司资料缺失不覆盖商品成功结果；批次中断后根据检查点跳过已完成商品和公司。

## 7. 验收标准

- 5 个商品都有明确终态，不崩溃、不静默空成功；
- 至少 4 个商品成功取得详情和 SKU；
- 至少 3 个不同公司成功取得 1688 官方主体资质；
- 同一 `memberId` 的公司页面只访问一次；
- 商品和公司关系数量可与输入样本核对；
- 每个关键非空字段有来源 URL 和采集时间；
- L0 原始证据可用于离线重放；
- 固定样本、去重、断点和批次契约测试通过。

## 8. 后续边界

本轮只完成 1688 和 5 个样本验证，不接入淘宝、京东、拼多多、抖音，不写正式数据库，也不部署 n8n。验证通过后，再把同一批次命令接入 n8n 的触发、重试、人工登录门禁和通知节点。
