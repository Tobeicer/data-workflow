# 1688 娃娃机公司全字段试采设计

日期：2026-07-13  
状态：设计已确认，纯解析与单商品浏览器试采均已完成  
上位基线：`../../数据工作流与游艺圈系统对接执行基线.md`

## 1. 目标

以一个 1688 商用娃娃机商品为样本，建立以下可追溯链路：

```text
商品 → 店铺 → 公司主体 → 工厂能力 → 主体资质与证据
```

先完成单商品低频验证，再按规范化店铺 URL 和主体键去重扩大公司队列。

## 2. 数据边界

允许采集普通登录用户可见的商品、店铺、工商主体、工厂、证书、专利和公开经营联系方式。

禁止获取身份证号、私人联系方式、家庭住址、订单、购物车、支付、聊天、后台数据，以及任何需要绕过登录、验证码、权限、签名或风控的内容。

## 3. 建模方式

商品、店铺、公司和工厂能力独立保存，通过带证据的关系关联。公司字段不得平铺到每个商品，也不使用外部企业 API 替代 1688 的工厂和经营来源事实。

### 商品 `source_product`

- `offer_id`、标题、URL、价格、销量文本和图片；
- 商品属性、SKU、库存和采集时间。

### 店铺 `source_shop`

- 店铺 ID、规范化 URL、名称、主营、地区和入驻信息；
- 带时间的经营与服务快照；
- 公开经营联系人、电话、邮箱、旺旺和地址。

### 公司 `source_company`

- 公司名称、统一社会信用代码、注册号；
- 法定代表人、注册资本、成立日期；
- 企业类型、登记机关、营业期限、年报时间；
- 注册地址、经营范围和主体资质页面 URL。

统一社会信用代码是最强主体键；没有代码时使用规范化公司名称和地址作为候选键并进入复核。

### 工厂能力 `source_factory_profile`

- 厂房面积、员工、设备、生产线和研发能力；
- 证书、专利、认证、经营模式、加工能力和服务保障；
- 所有变化型字段保存来源时间和口径。

### 关系

- 商品—店铺；
- 商品—公司候选；
- 店铺—公司主体。

关系保存 `relation_type`、来源页面、采集时间、匹配方法、置信度和冲突原因。不同名称不得互相覆盖。

## 4. 来源证据

每类页面保存最终 URL、页面类型、采集时间、原始 HTML/JSON、文本快照、必要截图和哈希。

字段级证据记录字段、值、来源页面、标签或结构化路径。L0 保存原始证据，L1 保存标准化实体，L2 保存关系、快照、冲突、质量和复核。

## 5. 采集流程

1. 低频取得一个商用娃娃机 `offer_id`。
2. 保存商品详情和店铺/公司入口。
3. 访问店铺、公司档案和工厂公开页面。
4. 使用商品页公开 `memberId` 访问 1688 官方主体资质页。
5. 保存 L0，再由纯解析模块生成 L1/L2。
6. 登录、验证码、限流或结构变化返回明确状态并停止扩大请求。

浏览器层只负责导航和原始证据；字段解析使用固定样本测试。

## 6. 输出

```text
data-workflow/runtime/runs/1688/<run_id>/
├─ l0/
├─ l1/
├─ l2/
├─ run_manifest.json
└─ run_result.json
```

L1 至少包含商品、店铺、公司、工厂、证书和专利；L2 至少包含三类关系、复核队列和质量报告。页面没有信息时保存明确缺失状态，不伪造字段或把未知写成 0。

## 7. 停止条件

- 登录失效或人工验证；
- 403/429、权限或签名限制；
- 页面没有公司入口；
- 店铺名、供应商名和证照主体名冲突；
- DOM/结构化字段变化；
- 请求明显影响目标服务。

## 8. 验收

- 至少取得 1 个商品及其店铺和公司关系；
- 保存关键页面原始证据；
- 页面公开的工商、工厂和联系方式字段可结构化；
- 关键非空字段可回溯到来源；
- 冲突、缺失和受限状态明确；
- 固定样本测试通过；
- 线上试采低频且不绕过风控。
