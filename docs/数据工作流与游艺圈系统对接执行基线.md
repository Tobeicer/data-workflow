1. 用户最新明确指令。

3. `docs/数据工作流总体技术设计.md`，负责跨来源技术架构、数据契约和数据库对接逻辑模型。
4. `docs/数据工作流建设路线图.md`，负责实施任务、依赖、状态、验收和下一执行点，不改变上位技术语义。
5. 当前来源的 `data-workflow/adapters/<source>/README.md`，负责来源特有边界、命令、能力和运行证据。
6. `docs/游艺圈游戏游艺设备完整分类清单.md`，负责分类、平台类目映射、关键词和包含/排除规则。

同一事实只在最高职责文档中定义。根 README 和 `AI_HANDOFF.md` 只提供入口，不复制业务规则、动态状态或执行过程。

## 2. 项目定位与已确认决策

- 游艺圈是游戏游艺设施商品与厂家信息供应平台，不是商城，不负责交易功能。
- 数据层目标是在公开或授权边界内建立可追溯的全量信息镜像，不代表绕过验证码、签名、权限、限流或平台规则。
- 1688、淘宝、京东、拼多多、抖音和闲鱼等大平台通过公开数据爬虫建立全量镜像，并分别建设独立采集适配器；漫立方等邀约入驻行业店铺优先使用授权数据 API。
- n8n 是共享控制面；Python/Node 和来源专用自动化是执行面。
- 共享的是触发、状态、锁、重试、质量、人工门禁、告警和交付契约，不强行共享不同平台的页面解析、登录、资质、字段或风控处理。
- 数据岗位尽可能完整保留来源字段、原值、证据和历史变化，不因当前平台缺少消费字段而裁剪 L0-L2。
- 跨平台采用“公共核心字段 + 类型化属性事实 + 原始全字段”的混合模型；缺失字段由平台按契约选择不展示，但数据侧保留缺失原因。
- 商品、SKU、店铺、公司、厂家角色、资质、图片和文档分别建模；店铺不默认等于厂家，跨平台相似记录只建立候选关系，不静默覆盖来源事实。
- 分类清单在当前阶段用于采集范围、平台类目映射、关键词覆盖和分类候选，不由数据侧直接决定平台最终展示分类。
- 数据岗位不设计、迁移或维护正式业务数据库，不得由数据侧直接写入正式产品库。
- 平台负责人通过已确认的 L3/API/权限隔离接收契约完成校验、审核、正式写库和渲染。

正式项目路径为：

- `data-workflow/orchestration/n8n/`
- `data-workflow/adapters/<source>/`
- `data-workflow/runtime/`
- `data-workflow/deliveries/`
- `legacy-workflow/`

## 3. 职责边界

### 数据负责人

- 来源发现、能力矩阵、公开/授权边界和停止条件。
- 六个大平台的独立爬虫适配器，以及邀约商户授权 API、文件和公开资料适配。
- L0 原始响应、页面、文件、图片、请求元数据、来源证据和哈希归档。
- L1 来源实体、完整来源字段、原值与标准值。
- L2 关系、快照、变化、字段观测、匹配候选、质量问题和复核队列。
- n8n 数据工作流、触发、锁、重试、熔断、人工门禁、告警和运行报告。
- 按双方确认的版本化契约生成可重建的 L3 交付。

### 平台负责人

- 正式数据库、表结构、迁移、索引、权限和生产维护。
- 正式业务 ID、平台枚举、审核状态、分类发布、搜索和字段渲染。
- 导入 API 或权限隔离的 `ingest/staging` 接收区。
- L3 校验、错误回执、审核、晋级、发布和回滚。
- 小程序、APP、Web 后台及交易相关能力。

### 双方共同确认

- L3 契约版本、核心字段、可扩展属性、长度、枚举和必填规则。
- 来源唯一键、交付幂等键、正式实体匹配和合并规则。
- 商品—SKU—店铺—公司—厂家关系及跨平台候选处理。
- 图片、视频、文档和对象存储引用方式。
- 分类候选、平台最终分类、错误回执、复核回传和回滚方式。

## 4. 数据分层与运行架构

| 层级 | 内容 | 约束 |
|---|---|---|
| L0 | 原始 JSON/HTML/CSV/XLSX/PDF/图片、请求响应元数据、来源链接、采集时间、批次和哈希 | 不覆盖，是重放和审计依据 |
| L1 | 来源商品、SKU、店铺、公司、厂家角色、资质、媒体、完整来源字段及标准化值 | 保留平台语义，不受当前正式库字段限制 |
| L2 | 关系、动态快照、属性事实、变化、实体匹配候选、分类候选、质量和复核 | 冲突、不确定性和缺失原因显式保存 |
| L3 | XLSX/CSV/JSONL/API 请求体、对象存储引用或约定接收包 | 可由 L0-L2 重建，不反向覆盖 L0-L2 |

```text
定时 / Webhook / 条件 / 人工触发
                  ↓
          n8n shared 控制面
                  ↓
       独立来源 workflow / adapter
                  ↓
             L0 → L1 → L2
                  ↓
       质量门禁 / 人工复核门禁
                  ↓
                 L3
                  ↓
       平台校验、审核、写库和发布
```

详细状态机、逻辑数据集、推荐索引和 L3 envelope/receipt 契约以 `docs/数据工作流总体技术设计.md` 为准。

## 5. 字段融合和实体原则

- 公共核心字段用于跨平台搜索、排序和基本展示。
- 平台特有字段进入带类型、单位、原始值、来源、证据、时间和可信状态的属性观测，不通过无限增加公共表列解决。
- 每个平台的原始字段名、字段路径和值完整保留；未知字段先保存，后续可重映射。
- `not_provided`、`not_accessible`、`not_applicable`、`parse_failed` 和 `unknown` 必须区分，不用空字符串或 `0` 伪造未知。
- 同一字段存在多来源或多时间值时保留全部观测；平台选择展示值不能删除其他证据。
- 原始标签语义不同的字段不得因名称近似而合并。例如 1688“工厂面积”和“厂房面积”必须分别映射为 `factory_area_sqm` 与 `factory_building_area_sqm`。
- 价格、库存、销量、供货状态等高频动态值使用带采集时间的快照，不只覆盖保存当前值。
- SKU 不简化成单一文本；规格组、规格值、价格、库存和图片关系分别保存。

## 6. 触发、运行和更新原则

支持以下触发：

| 触发方式 | 用途 |
|---|---|
| 定期/定时 | 全量基线、增量扫描、类目树和来源健康检查 |
| 自动触发 | 授权 API 事件、上游任务完成和依赖就绪 |
| 条件触发 | 新鲜度超限、结构变化、数量异常和质量下降 |
| Webhook | 入驻店铺同步、合作方文件或平台回执 |
| 人工任务 | 登录、验证码、异常补采和高风险复核 |

统一 `run_id` 为 `<source>_<mode>_<YYYYMMDD_HHMMSS>`。同一 `run_id` 和阶段重复执行应幂等，同一来源全量任务默认不并发。

采集分为首次全量基线和后续分层更新：价格库存高频轻量检查，商品详情按变化补抓，店铺低频更新，厂家资质独立低频更新，类目树和字段结构定期巡检。全量不等于每次重复下载所有不变详情和图片。

每次运行必须输出符合 `data-workflow/contracts/schemas/run_result.schema.json` 的 `run_result.json`；n8n 只解析机器契约，不解析自然语言日志。

## 8. 分类、关键词和采集边界

- 平台类目映射、平台专用关键词、包含/排除和待复核规则以分类清单为唯一分类参考。
- 范围判断使用 `included`、`excluded`、`review_required` 三态；无法判断不能直接归入 A14。
- A14 是真实“综合/主题/非标设备”分类，不是未知数据收容项。
- 分类映射和关键词必须带规则版本、平台、验证时间和证据；平台类目树变化需保存快照。
- 数据侧产生分类候选、置信度和理由，平台侧负责正式分类晋级和展示。

## 9. 正式数据库和 L3 对接边界

默认顺序：

1. 数据侧保存 L0-L2。
2. 数据侧按确认契约生成版本化 L3。
3. 平台优先通过内部导入 API 接收，其次使用权限隔离的 `ingest/staging`，L3 文件导入作为兜底。
4. 平台校验、审核并写入正式业务表。
5. 平台返回批次级和记录级错误回执；数据侧仅在来源解析错误时修正 L0-L2，否则更新 L3 适配。

禁止绕过平台校验直接写 `public.product`、`public.accessory`、`public.manufacturer` 等正式业务表。

数据库快照仅作历史/当前环境和对接契约参考：PostgreSQL `192.168.1.98:5432`、数据库 `postgres`、Schema `public`、Navicat 连接名 `youyiquan`。工作区当前没有 `database/` 和 `database/public.sql`，不得虚构快照存在；这些连接信息不是直接写正式业务表的授权。密码只进入未跟踪的 `.env.local` 或受控凭据，不得写入 Markdown。

平台源码和正式数据库定义尚未提供。`docs/数据工作流总体技术设计.md` 中的表/数据集是数据侧推荐逻辑模型，不是正式库迁移授权。

## 10. 质量、安全和启用门禁

每批至少检查总数、成功数、唯一数、重复率、字段覆盖、图片成功率、新增/修改/下架/无变化、来源可追溯性、结构漂移和复核数量。各平台按能力矩阵设定自己的必需字段，不能用 1688 的厂家字段覆盖率要求淘宝。

以下情况不得自动正常交付：解析为 0、数量异常下降、关键字段或图片成功率显著下降、结构/枚举变化、大量价格归零、异常下架、AI Schema 失败或成本超限。

遇到验证码、滑块、登录失效、403/429、权限不足、签名破解要求或非公开数据时停止当前路径，返回明确状态并转人工、降频或停用，不绕过限制。

来源只有同时具备以下证据，才能申请将 registry 的 `enabled` 改为 `true`：

1. 经过审查的来源 workflow JSON。
2. 受控凭据和登录态配置。
3. 严格离线 dry-run 与脱敏 fixture 验证。
4. 在线小样本质量证据、失败路由和人工门禁证据。
5. 全量/增量、恢复、幂等和交付回执验证。
6. 连续稳定运行验收。
