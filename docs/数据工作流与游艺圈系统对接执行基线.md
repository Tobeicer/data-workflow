# 数据工作流与游艺圈系统对接执行基线

版本：V1.2
日期：2026-07-14
状态：当前唯一执行基线

## 1. 权威级别

冲突时按以下顺序处理：

1. 用户最新明确要求。
2. 本文档。
3. 当前来源执行文档。
4. 总体方案或通用执行指南。
5. L3 消费兼容规范。
6. `docs/project-split/`、`docs/requirements/` 历史参考。

历史目录不得覆盖当前职责、数据分层、写库边界和执行顺序。

## 2. 当前决策

- 游艺圈是游艺设施商品与厂家信息供应平台，不是商城；数据工作流不负责交易功能。
- 当前阶段先全面推进稳定采集工作流；分类、字段渲染和正式数据库模型不属于数据负责人的当前实施重点。
- `data-workflow/` 是独立、可版本化的数据资产生产子项目。
- n8n 是控制面；Python/Node 脚本是执行面。
- 数据负责人生产完整、可追溯、可重放的 L0-L2 数据资产，并按平台契约生成 L3。
- 平台负责人维护正式数据库和平台应用，负责数据接收后的审核、晋级、发布与回滚。
- 平台接入优先使用内部导入 API，其次是约定的权限隔离接收区，L3 文件导入作为兜底。
- 平台源码只用于定稿 L3 适配，不得反向裁剪 L0-L2。
- “全网覆盖”通过来源登记体系执行：只登记合规公开或已授权来源，按来源分级、验证并逐来源稳定化，不承诺无边界或高频抓取。
- 更新机制按来源能力配置定时、自动或条件触发，目标是持续保持商品和厂家信息的新鲜度。
- 正式路径为 `data-workflow/orchestration/n8n/`、`data-workflow/adapters/<source>/`、`data-workflow/runtime/`、`data-workflow/deliveries/` 和 `legacy-workflow/`。

## 3. 职责边界

### 数据负责人

- 来源发现、登记、分级、评估、验证以及公开或授权边界记录。
- 负责漫立方、1688、淘宝、京东、拼多多、抖音和闲鱼七个平台的采集适配器，按来源能力设置定时、自动或条件触发更新。
- 公开数据、授权文件、API、网页、小程序和行业资料的采集或接收适配，尽可能完整保留来源字段。
- 原始响应、文件、图片、文档和来源证据归档。
- L0-L2 清洗、标准化、去重、分类候选、实体关系、快照、变化和质量处理。
- SKU、规格、价格、库存、供货状态等来源信息及历史快照。
- AI 辅助抽取、分类、相似判断和复核原因生成。
- n8n 编排、重试、状态、来源健康检查、更新检测、质量门禁和复核队列。
- 按双方确认的契约生成可替换的 L3 交付。

### 平台负责人

- 正式数据库、表结构、迁移、索引、权限和生产维护。
- 正式业务 ID、分类、标签、枚举、状态机、字段消费与渲染和业务约束。
- 内部导入 API、权限隔离接收区或文件导入器。
- L3 接收校验、错误回执、审核、正式记录晋级、发布和回滚。
- 小程序、APP、Web 后台、搜索、推荐和其他平台应用。

### 双方共同确认

- L3 契约版本、消费字段映射、字段类型、唯一键、幂等键和枚举。
- 来源合法性与授权边界、质量门槛、异常处置和变更兼容。
- 图片、文档和对象存储传输方式。
- 错误回执、复核回传、验收标准和回滚方式。
- 数据侧不得直接写正式业务表；平台新增消费需求优先通过 L3 映射兼容。

## 4. 数据分层

| 层级 | 内容 | 约束 |
|---|---|---|
| L0 原始来源层 | 原始 JSON/HTML/CSV/XLSX/PDF/图片、请求响应元数据、来源链接、采集时间、批次和哈希 | 不覆盖，是重放和审计依据 |
| L1 标准化来源资产层 | 厂家、产品、配件、SKU、规格、参数、图片、来源分类、价格和供货状态 | 尽可能完整保留来源字段和来源语义，不受当前平台字段限制 |
| L2 关系、快照与质量层 | 多对多关系、价格库存历史、分类映射、实体匹配、变化结果、质量和复核状态 | 冲突和不确定性必须显式保存 |
| L3 消费交付层 | XLSX、CSV、JSONL、API 请求体、对象存储引用或约定接收包 | 可随平台契约重建，不得反向覆盖 L0-L2 |

## 5. 运行边界

```text
触发
→ `data-workflow/orchestration/n8n/` 编排、状态、重试和人工门禁
→ `data-workflow/adapters/<source>/` 来源适配器与通用处理脚本
→ L0
→ L1
→ L2
→ 质量与人工门禁
→ L3
→ 平台导入、审核和发布
```

n8n 不承担大型 Excel、图片批处理、复杂浏览器脚本或大表清洗。脚本通过退出码和机器可读状态与 n8n 交互。

每个适配器最终输出 `run_result.json`，至少包含：

- `run_id`、`source`、`workflow_version`；
- `status`、`started_at`、`finished_at`；
- `counts`、`artifacts`、`quality_gate`；
- `review_required`、`retryable`；
- `error_code`、`error_message`。

## 6. 正式写库与系统对接

默认流程：

1. 数据负责人保存 L0-L2。
2. 数据负责人按确认契约生成 L3。
3. 平台接收、校验、审核并写入正式系统。
4. 平台回传错误、审核结果和缺失字段需求。
5. 数据负责人更新 L3 适配器；只有来源解析有误时才修改 L0-L2 处理。

禁止绕过平台校验直接写 `public.product`、`public.accessory`、`public.manufacturer` 等正式业务表。

系统源码到达后只核对：真实字段、类型、枚举、唯一约束、导入/审核流程、文件上传、幂等、错误回执和结果回传。产出是 L1/L2→L3 映射与消费适配器，不是重做来源模型。

## 8. 完成标准与当前重点

单个来源完成接入需具备：重复执行入口、L0-L2、更新策略、错误状态、恢复方式、测试样本、质量门禁和 L3 输出。

数据工作流基础体系完成需证明：多来源可独立启停、失败可发现和恢复、增量可解释、原始资产可重放、交付可复现，并连续稳定运行不少于 30 天。

当前已完成：

- 漫立方 3128 条商品和 5528 张图片交付。
- 1688 商品/SKU 样本与公司纯解析能力。
- 淘宝人工登录态搜索和详情补采原型。

当前执行顺序以逐来源稳定化为原则：

1. 完成文档、目录和统一契约治理。
2. 建立统一批次、状态和 `run_result.json`。
3. 将漫立方接入增量、无变化和失败恢复流程。
4. 完成 1688 公司浏览器试采器并接入统一编排。
5. 按淘宝、京东、拼多多、抖音、闲鱼的顺序完成小样本、契约、稳定化和 n8n 接入。
6. 收到平台契约后定稿 L3；分类、字段渲染和正式数据库模型由平台侧推进，不作为数据负责人当前实施重点。
