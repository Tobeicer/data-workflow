# 数据工作流与游艺圈系统对接执行基线

版本：V1.2
日期：2026-07-14
状态：当前唯一执行基线

## 1. 权威级别

冲突时按以下顺序处理：

1. 用户最新明确要求。
2. 本文档。
3. 当前来源指南（`data-workflow/<source>/`）。
4. `游艺圈数据导入字段规范_v2.md`（仅 L3 Excel 契约）。
5. `docs/project-split/`、`docs/requirements/` 历史参考。

同一规则只在最高优先级文档完整定义，其他文档使用引用。

## 2. 当前决策

- `data-workflow/` 是独立、可版本化的数据资产生产子项目。
- n8n 是控制面；Python/Node 脚本是执行面。
- 数据岗位生产完整、可追溯、可重放的 L0-L2 数据资产，并按平台契约生成 L3。
- 数据岗位不设计或维护游艺圈正式业务数据库，不直接写正式业务表。
- 平台技术同事负责正式表、业务 ID、枚举、导入、审核、晋级和发布。
- 平台接入优先使用内部导入 API，其次是约定的权限隔离接收区，L3 文件导入作为兜底。
- 平台源码只用于定稿 L3 适配，不得反向裁剪 L0-L2。
- 大平台是补充来源，不建设全量商品镜像库。
- 完整采集受公开/授权范围、平台规则、频率、账号稳定性、成本和业务用途约束。

## 3. 职责边界

### 数据岗位

- 来源发现、登记、评估、授权和合规边界记录。
- 公开数据、授权文件、API、网页、小程序和行业资料的采集或接收适配。
- 原始响应、文件、图片、文档和来源证据归档。
- 清洗、标准化、去重、分类候选、实体关系和质量处理。
- SKU、规格、价格、库存、供货状态等来源信息及历史快照。
- AI 辅助抽取、分类、相似判断和复核原因生成。
- 增量比较、来源健康检查、质量门禁、复核队列和交付包生成。
- n8n 数据工作流、重试、状态、人工门禁、告警和运行报告。

### 平台技术同事

- 正式数据库、表结构、迁移、索引、权限和生产维护。
- 正式业务 ID、分类、标签、枚举、状态机和业务约束。
- 导入 API、待审核区、错误回执、审核、晋级和发布。
- 小程序、APP、Web 后台、搜索、推荐、订单、支付和其他平台功能。

### 双方确认

- 消费字段映射和版本号。
- 唯一键、幂等键、分类和枚举。
- 图片、文档和对象存储传输方式。
- 错误回执、复核回传、验收标准和回滚方式。

## 4. 数据分层

| 层级 | 内容 | 约束 |
|---|---|---|
| L0 原始来源层 | 原始 JSON/HTML/CSV/XLSX/PDF/图片、请求响应元数据、来源链接、采集时间、批次和哈希 | 不覆盖，是重放和审计依据 |
| L1 标准化来源资产层 | 厂家、产品、配件、SKU、规格、参数、图片、来源分类、价格和供货状态 | 保留完整来源语义，不受当前平台字段限制 |
| L2 关系、快照与质量层 | 多对多关系、价格库存历史、分类映射、实体匹配、变化结果、质量和复核状态 | 冲突和不确定性必须显式保存 |
| L3 消费交付层 | XLSX、CSV、JSONL、API 请求体、对象存储引用或约定接收包 | 可随平台契约重建，不得反向覆盖 L0-L2 |

## 5. 运行架构

```text
定时 / Webhook / 人工 / 事件
              ↓
         n8n 控制面
              ↓
   来源适配器与通用脚本
              ↓
      L0 → L1 → L2
              ↓
     质量门禁 / 人工门禁
              ↓
            L3
              ↓
     平台导入、审核与发布
```

n8n 负责触发、依赖、锁、超时、重试、状态、人工确认和通知。Python/Node 脚本负责采集、浏览器自动化、清洗、图片、文档、AI、增量比较、质量计算和交付包生成。n8n 不承担大型 Excel、图片批处理、复杂浏览器脚本或大批量 AI 处理。

### 统一状态与脚本契约

每次任务使用唯一 `run_id`：`<source>_<mode>_<YYYYMMDD_HHMMSS>`。

正常状态：`created → prechecking → ready → collecting → raw_archived → normalizing → matching → enriching → validating → waiting_review / packaging → delivered → completed`

异常状态：`login_required`、`human_verification_required`、`rate_limited`、`source_changed`、`parser_drift`、`quality_rejected`、`retryable_failed`、`manual_stopped`、`permanently_disabled`

统一命令目标：

```powershell
python run_source.py precheck --config <config> --run-id <run_id>
python run_source.py collect --config <config> --run-id <run_id>
python run_source.py normalize --run-id <run_id>
python run_source.py diff --run-id <run_id> --baseline <baseline_run_id>
python run_source.py enrich --run-id <run_id>
python run_source.py validate --run-id <run_id>
python run_source.py package --run-id <run_id>
python run_source.py resume --run-id <run_id>
```

每个适配器最终输出 `run_result.json`，至少包含 `run_id`、`source`、`workflow_version`、`status`、`started_at`、`finished_at`、`counts`、`artifacts`、`quality_gate`、`review_required`、`retryable`、`error_code`、`error_message`。脚本通过退出码和机器可读状态与 n8n 交互；n8n 不解析自然语言日志。

## 6. 触发方式

| 触发 | 适用场景 | 约束 |
|---|---|---|
| Webhook/事件 | 合作方文件、厂家上传、平台反馈 | 必须有明确授权入口 |
| 定期 | 官网、政策、文审、价格库存、来源健康 | 不用高频轮询模拟实时 |
| 人工任务 | 关键词、厂家、产品、异常补采 | 记录任务人、范围和停止条件 |
| 人工授权 | 淘宝、1688 等登录来源 | 登录失效即暂停，不绕验证码 |
| API 按需 | 企业核验、付费数据 | 缓存、配额、有效期和成本上限 |

## 7. 来源定位

| 来源 | 当前定位 | 运行方式 |
|---|---|---|
| 漫立方 | 首个全量样板，后续做增量和周期校验 | 周期更新、人工全量复核 |
| 合作商/厂家授权数据 | 正式主数据优先来源 | 文件到达、Webhook、API、人工 |
| 1688 | 新品、参数、SKU、价格和供应商补充 | 人工登录、低频、专项任务 |
| 淘宝 | 参数、价格、关键词和同款候选 | 人工授权、低频专项 |
| 其他大平台（京东/拼多多/抖音/闲鱼） | 待逐个平台验证 | 小样本验证后决定启用、人工或停用 |
| 企业信息 API | 厂家主体候选核验 | 按需、缓存、配额和成本上限 |
| 政策/文审/行业资料 | 官方文档和知识资产 | 定期监控、原文归档、人工核验 |

## 8. 来源优先级

| 优先级 | 来源 | 用途 |
|---|---|---|
| P0 | 授权 Excel、后台导出、官方 API、合作方文件 | 正式主数据优先来源 |
| P1 | 厂家官网、公开网页和商品页 | 长期公开补充来源 |
| P2 | 普通用户可访问的小程序接口 | 验证公开范围后接入 |
| P3 | 抓包识别公开接口 | 用于接口发现和验证，不作为唯一依赖 |
| P4 | PDF、画册、图片和 OCR | 字段补充，必须复核 |
| P5 | 大平台和潜在线索 | 低频候选，不直接进入正式库 |

## 9. 单来源标准流程

```text
登记来源与用途
→ 确认公开/授权边界和停止条件
→ 小样本验证
→ 记录字段和接口
→ 批量采集或文件接收
→ L0 原始归档
→ L1 标准化
→ L2 去重、关系、快照、变化和质量
→ AI 辅助与人工复核
→ L3 交付
→ 更新计划和健康检查
```

## 10. 每次运行产物

| 产物 | 内容 |
|---|---|
| L0 | 原始响应、文件、图片、请求元数据、来源证据和哈希 |
| L1 | 完整来源字段的标准化实体和明细 |
| L2 | 关系、快照、变化、质量、冲突和复核队列 |
| L3 | 当前平台契约需要的 XLSX/CSV/JSON/API 请求体 |
| 状态 | `run_result.json`、清单、错误记录和质量报告 |

价格、库存、销量、供货状态必须带采集时间并保留历史。

## 11. 字段与关系原则

- 先保存来源原值，再生成标准值。
- 不用空字符串或 0 伪造未知信息。
- 公司、店铺、商品、SKU、图片、分类和文档分别建模，通过关系关联。
- 同一产品可有多个来源和供应商；同一配件可适配多个机型。
- 平台当前不接收的字段仍保留在 L0-L2。
- 分类、标签、实体合并和高风险结论保留置信度与复核状态。

## 12. 质量门禁

每批至少检查：总数、成功数、唯一数和重复率；必填、来源、时间、图片和分类完整率；新增、修改、下架、失败和无变化数量；原始与清洗记录可核对性；页面结构和字段枚举变化；低置信度、冲突和复核数量。

以下情况不得自动正常交付：数量异常下降或解析为 0；必填完整率、图片成功率或分类覆盖率显著下降；字段集合、页面结构或枚举变化；大量价格归零、异常下架或 AI Schema 失败；付费 API 或 AI 成本超预算。

## 13. 停止与降级

遇到验证码、滑块、403/429、登录失效、权限不足、签名破解要求、来源异常或非公开数据时停止当前路径。

可降级为：人工登录、人工任务、小样本、文件导入、只保存页面可见信息或暂停来源。不得为了完成数量绕过平台限制。

## 14. AI 使用规则

AI 只处理确定性规则难以稳定解决的任务：非结构化抽取、分类建议、相似候选、图片描述、缺失和复核原因。

无人值守输出必须包含：任务版本、来源标识、结构化结果、置信度、理由、缺失、冲突和复核标记。

以下结论必须人工确认：工商主体和认证、文审/法律结论、侵权判断、低置信度实体合并、高风险联系方式和可能导致下架或信用处罚的结论。

模型、提示词和 Schema 需要版本号；相同输入使用哈希缓存；设置任务和周期成本上限。

## 15. 稳定性与安全

- 同一 `run_id` 和阶段重复执行应幂等。
- 同一来源全量任务默认不并发。
- 网络超时和临时 5xx 可指数退避重试。
- 登录失效、验证码、权限不足和字段变化不盲目重试。
- 长任务按分类、分页、关键词或商品 ID 保存检查点。
- 连续 403/429、验证码或高失败率时立即熔断，保存现场并转人工或停用。
- 密钥、Cookie、数据库密码和登录态只进入安全凭据或未跟踪环境文件。
- 开发、测试、常驻环境使用不同配置和凭据。

## 16. 正式写库与系统对接

默认流程：

1. 数据岗位保存 L0-L2。
2. 数据岗位按确认契约生成 L3。
3. 平台接收、校验、审核并写入正式系统。
4. 平台回传错误、审核结果和缺失字段需求。
5. 数据岗位更新 L3 适配器；只有来源解析有误时才修改 L0-L2 处理。

禁止绕过平台校验直接写 `public.product`、`public.accessory`、`public.manufacturer` 等正式业务表。

## 17. 标准交付包

```text
delivery_<source>_<batch>/
├─ README.md
├─ manifest.json
├─ schema.json 或字段字典.xlsx
├─ normalized_records.*
├─ relations/
├─ snapshots/
├─ images/
├─ documents/
├─ change_summary.*
├─ review_queue.*
├─ quality_report.*
└─ checksums.sha256
```

交付说明必须包含来源、批次、方式、时间、唯一键、字段、变化数量、质量、复核、文件引用、已知限制和更新频率。

## 18. 更新与交付前确认

- 批次、来源、唯一键和字段字典明确。
- 质量报告、变化摘要和复核队列齐全。
- 图片和文档引用可访问。
- L3 可从 L0-L2 重建。
- 平台接收方式和错误回执已确认。

## 19. 小程序公开接口发现

仅适用于普通微信用户正常浏览可见的分类、商品列表、详情、SKU、价格、库存、图片和参数。

可行性条件：正常会话能复现分类/列表/详情接口；分页字段清晰；有稳定商品 ID 或可解释的去重键；请求不依赖后台权限或破解签名。

发现顺序：分类树 → 商品列表与分页 → 商品详情 → SKU/规格与价格库存 → 图片和详情内容。

遇到 `cart`、`order`、`member`、`user`、`address`、`payment`、`admin`、`backend` 路径时停止并确认是否属于非目标数据。具体来源的命令只写入对应来源指南。

## 20. 监控与告警

监控任务耗时、阶段数量、来源健康、登录有效期、配额、成本、字段变化、磁盘和最近成功时间。

告警分级：信息（完成或无变化）、警告（少量失败、完整率下降、登录即将过期）、高（验证码、结构变化、解析为 0、质量门禁失败）、紧急（凭据泄露、异常高频请求或重大合规风险）。

## 21. 建设顺序

1. 完成文档、目录和统一契约治理。
2. 建立 n8n 基础编排。
3. 将漫立方接入增量、无变化和失败恢复。
4. 完成 1688 多品类工作流并接入统一编排。
5. 接入淘宝及其他授权/官方来源。
6. 逐个平台验证其他补充来源。
7. 连续运行不少于 30 天，验证告警、恢复、重放和可复现交付。

## 22. 完成标准

单个来源完成接入需具备：重复执行入口、L0-L2、更新策略、错误状态、恢复方式、测试样本、质量门禁和 L3 输出。

数据工作流基础体系完成需证明：多来源可独立启停、失败可发现和恢复、增量可解释、原始资产可重放、交付可复现，并连续稳定运行不少于 30 天。
