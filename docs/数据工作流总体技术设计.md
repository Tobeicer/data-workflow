# 游艺圈数据工作流总体技术设计

版本：V1.0

日期：2026-07-15

状态：已确认架构，待用户复核成文准确性

上位基线：`docs/数据工作流与游艺圈系统对接执行基线.md`

## 1. 文档职责

本文是数据工作流唯一跨来源技术设计，集中定义：

- n8n shared 控制面与每个平台独立适配器的边界；
- L0-L3、来源实体、类型化字段、证据、变化和复核模型；
- 数据侧建议的逻辑数据集、键和索引；
- L3 manifest、记录 envelope 和平台 receipt；
- 分类附录在采集、过滤和候选分类中的使用方式；
- 全量、增量、失败恢复、质量和长期稳定性设计；
- 当前契约漂移及进入实施前必须关闭的问题。

本文不保存来源命令、具体批次、动态来源状态或实施过程。来源命令和当前能力只写入对应 adapter README；实施任务、负责人、进度和验收只写后续唯一建设路线图。

本文中的表和索引是数据侧推荐逻辑模型，不是正式库迁移授权。平台负责人仍负责正式数据库、DDL、迁移、业务 ID、审核和发布；数据侧不得直接写入正式业务表。

## 2. 设计目标与非目标

### 2.1 目标

1. 在公开或授权边界内，持续获取游艺设施商品、SKU、店铺、公司、厂家角色、资质、媒体和变化数据。
2. 支持 1688 的丰富厂家字段、淘宝的稀疏厂家字段、京东受人工验证约束的资质字段，以及其他平台完全不同的字段与风控模型。
3. 保留每个平台原始全字段，同时提供可跨平台搜索、比较和对接的公共核心字段。
4. 首次建立可证明的全量基线，后续使用分层增量和定期重扫保持新鲜度。
5. 失败可发现、可停止、可恢复；受限页面不能被写成空成功或下架。
6. 所有标准化值、合并候选、分类候选和资质结论可追溯到 L0 证据。
7. L3 可幂等交付，平台能返回批次级、记录级和匹配/复核级回执。

### 2.2 非目标

- 不绕过验证码、滑块、签名、登录、权限或限流。
- 不承诺获取非公开字段，也不把公开边界之外的数据算入“全量”。
- 不把所有平台强制解析成相同字段集合。
- 不把店铺、公司、厂家、品牌方和供应商简化为同一实体。
- 不由数据侧创建或修改 `public.product`、`public.accessory`、`public.manufacturer` 等正式表。
- 不在 n8n Code 节点处理大型表格、图片、浏览器脚本或批量 AI。
- 当前不定稿平台前端渲染；本文只提供可供渲染的契约和规则。

## 3. 核心架构决策

采用“共享控制面、独立来源适配器”的结构：

```text
统一触发、状态、锁、重试、质量、人工门禁、告警、回执
                             ↓
                    n8n shared 控制面
                             ↓
 ┌────────┬────────┬────────┬────────┬────────┬────────┬────────┐
 │漫立方  │ 1688   │ 淘宝   │ 京东   │拼多多  │ 抖音   │ 闲鱼   │
 │独立API │独立爬虫│独立爬虫│独立爬虫│独立爬虫│独立爬虫│独立爬虫│
 └────────┴────────┴────────┴────────┴────────┴────────┴────────┘
                             ↓
                    L0 → L1 → L2 → L3
```

共享控制面只传递 `run_id`、请求、状态、指标摘要和资产引用。来源适配器独立拥有：

- 平台登录、Cookie/profile 和授权处理；
- 类目树、关键词、分页、商品 ID 和店铺发现；
- 列表、详情、SKU、图片、店铺、公司和资质采集；
- 平台特有结构解析、字段名、错误识别和检查点；
- 平台允许范围内的频率、停止和恢复策略。

这样既能复用稳定的运行治理，又不会把 1688 的厂家模型强加给淘宝或京东。

## 4. 平台能力矩阵

每个来源在开发前必须建立版本化能力矩阵。质量门禁按来源能力计算，不能要求所有平台具有 1688 同等厂家字段。

| 来源 | 商品/SKU | 店铺 | 公司/厂家 | 资质 | 主要约束 | 设计定位 |
|---|---|---|---|---|---|---|
| 漫立方 | 已有完整批次 | 来源方信息 | 当前主体仍需按证据核验 | 正式授权 API 待提供 | 当前链路含人工抓包 | 离线 replay 与 L0-L3 参考 |
| 1688 | 丰富 | 丰富 | 公司、工厂、供应商信息丰富 | 公开主体和平台资质较丰富 | 登录、403/429、滑块 | 丰富字段和厂家关系参考 |
| 淘宝 | 商品与详情可采 | 店铺可采 | 厂家通常稀疏，不能由店铺推断 | 通常有限 | 登录、风控、结构变化 | 稀疏字段参考 |
| 京东 | 商品与店铺可采 | 店铺可采 | 部分店铺可见主体 | 资质入口可能反复滑动验证 | 人工验证频繁 | 人工门禁参考 |
| 拼多多 | 商品为主 | 店铺信息依公开页面 | 厂家字段可能稀疏 | 依公开能力 | 类目扁平、标题污染 | 二次范围判断参考 |
| 抖音 | 商品与店铺依公开能力 | 可变 | 厂家字段不稳定 | 依公开能力 | 类目变化、风控 | 类目快照参考 |
| 闲鱼 | 二手刊登为主 | 卖家/店铺语义不同 | 厂家通常不能直接确认 | 非核心 | 弱类目、强文本、状态波动 | 文本与人工复核参考 |

能力矩阵中的字段状态使用：

- `supported`：稳定公开或授权获取；
- `partial`：只覆盖部分商品/店铺；
- `gated`：需要人工登录、验证或审批；
- `unavailable`：公开/授权路径不存在；
- `unknown`：尚未完成小样本验证。

矩阵变更必须带来源、版本、验证时间和证据批次。

## 5. 单平台独立执行结构

每个平台 adapter 最终拆分为以下明确职责，不实现一个不可恢复的巨大脚本：

| 模块 | 输入 | 输出 | 说明 |
|---|---|---|---|
| `precheck` | 来源配置、mode、run_id | 环境与能力结果 | 登录、凭据、磁盘、依赖、字段/类目版本 |
| `discover` | 类目/关键词计划 | 分类、关键词、店铺和商品种子 | 记录覆盖范围和分区 |
| `product-list` | 分区/分页检查点 | 来源商品索引 | 低成本变化发现 |
| `product-detail` | 商品 ID 队列 | 详情、参数、SKU | 只有需要时抓详情 |
| `shop` | 店铺 ID 队列 | 店铺来源记录 | 不推断工商主体 |
| `company` | 公司/资质入口 | 公司、工厂、资质证据 | 可与商品更新不同频率 |
| `media` | 媒体 URL/引用 | 哈希媒体和关系 | 内容寻址去重，关系不丢失 |
| `normalize` | L0 与来源映射 | L1 逻辑实体 | 原值与标准值并存 |
| `diff` | 当前与 accepted baseline | L2 变化和快照 | 失败运行不推进 baseline |
| `validate` | L0-L2 | 质量结果和复核队列 | 来源能力感知 |
| `package` | 通过门禁的 L0-L2 | L3 交付 | 版本化、幂等、可回执 |
| `resume/repair` | 检查点、问题范围 | 局部补采结果 | 只补失败分区 |

资质链路必须与高频商品更新链路解耦。京东等平台资质页遇到滑块时，商品更新可以按批准策略继续或暂停，但资质链路必须返回 `human_verification_required`，不能自动解验证。

## 6. 数据分层与目录

建议新运行统一为：

```text
data-workflow/runtime/runs/<source>/<run_id>/
├─ run_request.json
├─ run_manifest.json
├─ run_result.json
├─ stages/
│  └─ <NN>_<stage>/
│     ├─ stage_request.json
│     ├─ stage_result.json
│     └─ artifacts.json
├─ checkpoints/
├─ review/
├─ l0/
├─ l1/
└─ l2/
```

每个阶段的幂等单位是：

```text
run_id + stage + partition + input_fingerprint
```

只有当阶段状态成功、输入指纹一致且产物哈希仍有效时，恢复运行才允许跳过该阶段。所有状态文件先写临时文件，再原子替换。

## 7. 混合字段模型

不采用超宽表，也不采用不可治理的纯 JSON/EAV。统一采用三部分：

### 7.1 公共核心字段

用于跨平台搜索、排序、基本展示和关系：

- 来源、来源范围、来源记录 ID、来源 URL；
- 商品原始名、标准化名、品牌候选、型号候选；
- 来源状态、首次发现、最后发现和采集时间；
- 店铺、公司、厂家角色候选；
- 来源类目、标准分类候选；
- SKU、价格区间、供货状态和主媒体摘要；
- 质量、可信度和复核状态。

### 7.2 类型化属性事实

平台特有和长尾字段进入类型化属性事实。每条观测包含属性字典、原始标签、原始值、标准值、单位、来源、时间、规则版本、可信度和证据。

同一厂家可具有 `factory_area_sqm`、`factory_building_area_sqm`、`production_equipment_count` 等不同事实；其他平台没有时不伪造。前端可以有值才展示，但数据层仍保存缺失原因。

### 7.3 原始全字段

L0/L1 保存每个平台原始全字段、原始字段路径和未知扩展。尚未进入属性字典的字段也必须保留，以便不重新采集即可完成后续映射。

## 8. 缺失、冲突和显示规则

### 8.1 缺失状态

对平台和前端可简化显示，但数据层必须区分：

| 标准状态 | 含义 |
|---|---|
| `present` | 有有效观测值 |
| `not_provided` | 来源没有展示或明确未提供 |
| `not_accessible` | 当前公开/授权路径不可访问 |
| `not_applicable` | 对该实体不适用 |
| `not_requested` | 本次 mode 没有请求该字段 |
| `parse_failed` | 来源疑似有值，但解析失败 |
| `source_changed` | 页面/接口结构已变化 |
| `normalization_failed` | 原值存在但标准化失败 |
| `conflict` | 同一语义字段存在互相冲突的观测 |
| `redacted` | 因敏感或权限策略脱敏 |
| `unknown` | 现有证据无法进一步判断 |

未知不能写空字符串、`0`、`NULL`、`无` 或 `-`。没有 SKU 结构与 SKU 采集失败、没有图片与图片下载失败、来源未展示与本次未请求必须分别表达。

### 8.2 冲突选择

1. 先保存全部来源 claim，再生成当前选择值。
2. 不同原始语义不构成冲突。例如“工厂面积”和“厂房面积”分别保存。
3. 同一字段多值产生质量问题，保留全部证据。
4. 当前选择值必须记录 `selected_observation_id`、策略版本、理由和置信度。
5. 不设全局来源优先级，按字段语义选择：工商主体优先官方/主体资质；价格库存使用对应来源最新成功快照；市场销量保持各平台独立。
6. 低置信度实体合并、工商主体和认证结论必须人工或平台确认。

### 8.3 展示原则

- 公共端有值才展示，未提供字段不占空位。
- 价格、库存、销量、资质和厂家能力显示来源及截至时间。
- 未核验信息标注“来源平台披露”，不能渲染为游艺圈认证。
- 后台可查看缺失原因、冲突观测、证据和选择策略。
- 来源扩展可放“其他来源参数”；只有治理后的属性才能用于跨平台筛选和比较。

## 9. 实体边界与关系

必须遵循 `shop ≠ company ≠ manufacturer`：

- `shop` 是平台账号或经营店铺；
- `company` 是来源展示的工商主体；
- `manufacturer` 是公司或店铺相对于商品承担的厂家角色；
- `supplier`、`distributor`、`brand_owner` 是其他业务角色；
- 同一公司可有多个店铺，一个店铺也不能仅凭名称自动绑定公司；
- 每个平台商品是独立来源刊登，跨平台“同款”只建立匹配候选。

关系至少支持：

```text
published_by
sold_by
supplied_by
manufactured_by
brand_owned_by
legal_entity_of
authorized_distributor_of
compatible_with
```

关系必须带来源、有效时间、匹配方法、置信度、复核状态、冲突原因和证据引用。

## 10. 数据侧推荐逻辑数据集

以下可以实现为 JSONL、Parquet、分析库或权限隔离 staging。它们不是对平台正式表名的强制要求。

### 10.1 运行和来源

#### `source_definition`

关键字段：`source`、`source_type`、`authorization_mode`、`adapter_version`、`key_strategy_version`、`capability_matrix_version`、`boundary_ref`。

唯一键：`source`。

#### `collection_run`

关键字段：`run_id`、`source`、`mode`、`baseline_run_id`、`workflow_version`、`adapter_version`、`config_hash`、`trigger_type`、`status`、`started_at`、`finished_at`、`checkpoint_ref`、`run_result_ref`。

唯一键：`run_id`。请求幂等键为 `source + mode + requested_scope_hash + scheduled_window`。

建议索引：`(source, started_at desc)`、`(status, started_at)`、`baseline_run_id`。

### 10.2 来源实体

#### `source_product`

语义是平台或授权来源中的一条独立商品刊登，不等于游艺圈正式产品。

关键字段：`source_product_uid`、`source`、`source_scope`、`source_product_id`、`source_product_code`、`shop_uid`、`name_raw`、`name_normalized`、`source_url`、`source_status`、`brand_candidate`、`model_candidate`、`first_seen_at`、`last_seen_at`、`raw_record_ref`、`content_hash`、`key_strategy_version`。

唯一键：`UNIQUE (source, source_product_id)`；来源存在多个命名空间时扩为 `(source, source_scope, source_product_id)`。标题、价格和标准化名称不能作为强唯一键。

建议索引：店铺+状态、品牌、型号、最后发现时间、名称全文/trigram。

#### `source_sku`

关键字段：`source_sku_uid`、`source_product_uid`、`source_sku_id`、`option_signature`、`source_status`、`first_seen_at`、`last_seen_at`、`key_strategy_version`。

唯一键优先为 `(source_product_uid, source_sku_id)`；没有来源 SKU ID 时使用排序后的规格组/规格值签名加键版本。规格签名改变时生成新 SKU，不原地伪装成同一 SKU。

#### `sku_option`

关键字段：来源规格组/值 ID、来源名称、标准属性键、标准值和顺序。一个 SKU 对应多条 option，不简化为一段文本。

#### `source_shop`

关键字段：`shop_uid`、`source`、`source_shop_id`、`member_id`、原始/标准名称、URL、地区、状态、首次/最后发现时间。

唯一键：来源+稳定店铺 ID/member ID；店铺名只能是候选匹配信号。

#### `source_company`

关键字段：`source_company_uid`、`source`、`source_company_id`、工商原名/标准名、统一社会信用代码、注册号、法定代表人、注册资本、成立日期、注册地址、经营范围、登记状态和核验状态。

来源内唯一键：来源+来源公司 ID。只有格式校验并有证据的统一社会信用代码可作为跨来源强匹配键；名称、电话或地址单独不能自动合并。

#### `entity_relation`

关键字段：左右实体类型/UID、关系类型、来源、有效期、方法、置信度、复核状态、冲突原因和证据。

幂等键：左右 UID + 关系类型 + 来源 + 有效期/证据哈希。建议建立左右端点双向索引。

### 10.3 属性、资质、证据和媒体

#### `attribute_definition`

关键字段：`attribute_code`、namespace、显示名、语义定义、类型、单位、基数、允许值、适用分类、是否可筛选/比较、敏感级别和字典版本。

#### `attribute_observation`

关键字段：实体类型/UID、属性码、来源字段名、原始值、typed value、原始/标准单位、值状态、缺失原因、观测时间、规则/模型版本、置信度、复核状态和证据引用。

幂等键：实体 + 属性 + 来源 + 观测时间 + 原值哈希 + 证据哈希。

建议索引：`(entity_uid, attribute_code, observed_at desc)`；只给已治理可筛选字段建立数值/文本索引，不能对所有任意 JSON 字段盲目索引。

#### `qualification_claim`

关键字段：主体、资质类型、来源原名、证书号、颁发方、有效期、范围、来源声明状态、核验状态、方法、观测时间和证据。

核验状态：`source_asserted`、`unverified`、`issuer_verified`、`platform_verified`、`rejected`、`expired`、`revoked`。只有核验状态满足平台规则时才能展示认证徽章。

#### `platform_signal_snapshot`

保存“超级工厂、金牌制造、店铺等级、回头率、履约率”等平台时效信号，不将其自动映射为游艺圈认证等级。

#### `source_evidence`

关键字段：`evidence_id`、SHA-256、存储 URI、媒体类型、字节数、来源 URL、请求指纹、HTTP 状态、采集时间、run_id 和脱敏状态。

内容哈希唯一；同一内容可以通过关联记录保留多个来源 URL。

#### `evidence_locator`

把属性或结论定位到 JSON Pointer、字段路径、CSS/XPath、页面区域或原始标签。一个观测可引用多个 locator。

#### `media_asset` 与 `entity_media`

媒体按内容 SHA-256 去重；关系保留实体、角色、顺序、主图标志、来源端点、字段路径和观测时间。角色为 `main/gallery/detail/parameter/sku/unknown`。

### 10.4 动态快照和变化

#### `metric_snapshot`

高频动态数据的逻辑总类，实际可拆为价格、库存/供货、销量/热度表。

关键字段：商品/可选 SKU、指标类型、规范数值、原始文本、币种/单位、阶梯条件、观测时间和证据。

金额在 JSON 使用 decimal 字符串，平台落库使用定点 `numeric`，不使用二进制浮点。未知价格、库存和销量不写 `0`。

建议索引：`(entity_uid, metric_type, observed_at desc)`；大规模历史按来源和月份分区，可使用时间 BRIN。

#### `change_event`

类型：`created`、`updated`、`explicitly_delisted`、`suspected_missing`、`relisted`、`unchanged`、`retracted`。

本次未发现只能产生 `suspected_missing`。只有来源明确下架或可信完整扫描确认后，才能产生 `explicitly_delisted`。

### 10.5 匹配、分类、质量和复核

#### `entity_match_candidate`

关键字段：实体类型、左右 UID、matcher 及版本、输入快照哈希、分数、特征、强信号、冲突信号、决定状态、理由、复核人与时间。

唯一键：排序后的实体对 + 实体类型 + matcher 版本 + 输入哈希。单一标题、公司名或店铺名不得自动合并。

#### `classification_candidate`

关键字段：实体、来源类目、分类清单版本、候选分类码、候选类型、置信度、规则版本、范围三态、复核状态、理由和证据。

主分类只能有一个已确认值；适配机型、设备形态、玩法和运营属性可多值。无法映射不能直接归入 A14。

#### `quality_rule_result`

关键字段：run、规则集版本、规则 ID、作用域、实际/预期值、结果、严重级别和证据。

#### `quality_issue`

使用稳定 issue fingerprint 去重，保存实体、属性、问题类型、严重级别、实际值、原因码、状态、首次/最后发生和次数。

#### `review_item`

关键字段：问题、复核类型、实体、字段路径、优先级、候选值、建议动作、状态、负责人、创建/解决时间、结论和证据。

未关闭任务按 issue fingerprint 条件唯一；建议索引 `(status, priority, created_at)`。

### 10.6 交付和回执

#### `delivery_batch`

记录 contract/version、delivery_id、source、run、baseline、mode、幂等键、数据集、质量门禁、复核摘要、限制和 checksums。

#### `delivery_receipt`

记录平台 receipt_id、批次状态、接收/通过/拒绝/重复/待复核数量、Schema 错误、逐记录结果、重试建议、平台实体 ID 映射和匹配/分类反馈。

批次与记录回执都必须支持幂等重放。

## 11. 推荐索引原则

若平台接收区使用 PostgreSQL，建议但不授权数据侧直接创建：

- 来源商品：来源+来源范围+来源 ID 唯一；名称全文/trigram 只用于搜索，不唯一。
- SKU：商品+来源 SKU ID 唯一；无 ID 时商品+规格签名+键版本唯一。
- 店铺/公司：来源稳定 ID 唯一；已核验统一社会信用代码可建条件唯一索引。
- 证据和媒体：SHA-256 唯一。
- 属性观测：实体+属性+时间倒序；仅治理后的 typed value 建索引。
- 快照：实体+时间倒序，按来源/月分区，时间序列可用 BRIN。
- 原始 JSON：只给明确查询路径建立有限 GIN，不做全键无差别索引。
- 匹配：实体对+matcher 版本唯一，状态+分数倒序。
- 复核：状态+优先级+创建时间。
- 质量问题：issue fingerprint 唯一。
- 关系：左右实体+关系类型建立双向索引。

正式索引必须基于平台真实查询、数据量和执行计划验证，不因本文建议直接进入生产。

## 12. 分类清单附录的工程化

### 12.1 附录 A：平台类目映射

映射记录必须包含平台、来源类目 ID/路径、类目树快照版本、候选标准分类、置信度、规则版本、验证时间和证据。平台一个类目可映射多个候选，需结合标题、属性和详情继续判断。

### 12.2 附录 B：关键词库

使用“标准概念 → 平台专用搜索词 → 同义词 → 排除词 → 最近验证时间 → 有效率 → 启用状态”的结构。不能把全平台词简单去重后在每个平台全部执行。

需要建立覆盖台账：来源、分类、关键词、分区、已扫描页数、唯一商品数、有效率、最后检查点和最近成功时间。附录当前尚未覆盖的 A02-A14、F 范围必须通过运行证据逐步扩充。

### 12.3 附录 C：包含、排除和待复核

范围只使用：

- `included`
- `excluded`
- `review_required`

无法确定时进入复核，不生成正式分类；A14 只用于证据确认的综合/主题/非标设备。

规则执行结果必须带规则版本、命中条件、原因和证据。人工结论反向形成新的规则测试，但不能修改历史 L0。

## 13. n8n shared 控制面

### 13.1 Master 工作流

固定链路：

```text
触发入口
→ 请求规范化和去重
→ registry 与运行时熔断检查
→ 原子来源/profile/资源锁
→ 来源 workflow
→ shared stage runner
→ result/schema/产物校验
→ 状态路由
→ 质量或人工门禁
→ L3/平台回执
→ 完成、释放锁和告警
```

Master 不包含平台解析代码，也不携带 L0 正文、大图片或大型表格。

### 13.2 Shared 子工作流

| 子工作流 | 职责 |
|---|---|
| request normalizer | 白名单化 source/mode/stage，禁止 webhook 参数直接拼接 shell |
| trigger deduplicator | 处理定时/Webhook 重放，返回已有 run_id |
| lock manager | 原子获取、心跳、租约、hold、释放和过期审计 |
| stage runner | 调用 adapter 统一 CLI 或受控 runner |
| result validator | 校验 Schema、状态一致性、产物存在和哈希 |
| status router | 统一状态路由，来源不得自定义控制语义 |
| retry breaker | 阶段级有限重试、熔断和半开小样本探测 |
| review gate | 建立复核任务，记录 actor、时间、理由和恢复点 |
| quality gate | 读取执行面质量结果，不在 n8n 批量计算 |
| alert router | 分级、去重、升级和恢复通知 |
| receipt handler | 登记 L3 平台回执和 ID 映射 |
| stale run reconciler | 处理失去心跳的运行、过期锁和长期未成功来源 |

### 13.3 锁

至少需要：

- `source:<source>:write`：同来源 full/incremental/replay 写运行互斥；
- `profile:<source>`：真实浏览器 profile 单进程；
- `resource:<runner>:browser`：限制主机浏览器并发。

锁记录包含 run_id、n8n execution ID、mode、获取时间、心跳和租约到期。多 worker 时不能只使用普通 JSON 文件；必须使用具备原子 compare-and-set 与租约的 n8n 运营存储、独立控制面数据库或 Redis，且不得使用正式 `public` 业务表。

人工等待期间释放物理租约但保留来源 hold；恢复时重新 precheck 并重新获取锁。

### 13.4 重试和熔断

- adapter 内只允许少量请求级超时/临时 5xx 重试；
- n8n 只负责阶段级恢复；
- 两层共享总预算，避免重试乘法；
- 403/429、验证码、滑块、权限、签名和 parser drift 不盲目重试；
- rate limit 遵守 Retry-After 或冷却策略，之后只允许 half-open 小样本；
- schema 不合法或产物缺失视为契约故障，冻结后续阶段并高等级告警。

## 14. 统一状态与路由

正常状态：

```text
created → prechecking → ready → collecting → raw_archived
→ normalizing → matching → enriching → validating
→ waiting_review / packaging → delivered → completed
```

异常状态与动作：

| 状态 | 控制动作 |
|---|---|
| `retryable_failed` | 总预算内退避，从检查点恢复；耗尽后人工处理 |
| `rate_limited` | 停止、冷却、half-open 小样本，不推进水位 |
| `login_required` | 不自动重试；人工登录后重新 precheck |
| `human_verification_required` | 脱敏保存页面/截图，禁止自动解验证，进入人工门禁 |
| `source_changed` / `parser_drift` | 打开熔断器，冻结交付和 baseline，保留现场 |
| `quality_rejected` | 禁止 package/delivery，进入质量复核 |
| `manual_stopped` | 新人工请求前不得继续 |
| `permanently_disabled` | Master 在执行前拒绝所有采集触发 |

`partial_success` 不是可自动交付状态，必须经质量规则映射为 `waiting_review`、`quality_rejected` 或 `retryable_failed`。`blocked` 必须细分具体原因。失败、受限或部分遍历不得产生下架结论。

## 15. `run_result.json` 目标契约

当前 Schema 过于宽松，实施时应升级为严格结构，至少包括：

```text
run_result_schema_version
run_id
source
source_scope
workflow_version
adapter_version
config_hash
mode
stage
baseline_run_id
status
outcome
started_at
finished_at
counts_by_dataset
artifacts[]
quality_gate
review
retry
error
checkpoint
extensions
```

`artifacts[]` 包含 layer、dataset、URI、schema_version、record_count 和 SHA-256。`review` 包含 required、任务数和 queue URI；`retry` 包含 retryable、retry_after 和恢复阶段；`error` 包含 code、category、stage、message 和 details_ref。

核心对象关闭任意额外字段；来源扩展只能进入 `extensions`。所有 adapter 必须通过 contract tests，n8n 必须先校验再路由。

错误码使用类别前缀：`AUTH_*`、`ACCESS_*`、`SOURCE_*`、`PARSER_*`、`DATA_*`、`QUALITY_*`、`SYSTEM_*`、`DELIVERY_*`。自然语言日志不能代替错误码。

## 16. L3 交付契约

### 16.1 Manifest

`delivery_batch` 至少包含：

```text
contract_name
contract_version
delivery_id
delivery_mode             # full / delta / repair
source
source_scope
run_id
baseline_run_id
generated_at
producer
workflow_version
taxonomy_version
attribute_dictionary_version
resolution_policy_version
idempotency_key
datasets[]
quality_gate
review_required
review_summary
checksums_manifest
known_limitations[]
```

每个 dataset 声明名称、实体类型、Schema 版本、格式、路径/端点、record_count、业务键、幂等键、SHA-256、字节数和 `snapshot/upsert_delta` 操作模式。

### 16.2 Record envelope

每条记录统一：

```text
contract_version
dataset
schema_version
record_key
idempotency_key
operation                 # upsert / deactivate / retract
source
run_id
observed_at
payload_hash
payload
lineage
quality
```

重发相同 `idempotency_key + payload_hash` 必须返回 duplicate/no-op；相同幂等键但 payload hash 不同必须返回冲突错误，不能静默覆盖。默认不发送 hard delete。

### 16.3 平台错误回执

`delivery_receipt` 批次级至少包含 contract/version、delivery/receipt ID、idempotency key、接收/完成时间、状态、received/accepted/rejected/duplicate/pending_review 数量、Schema 错误、逐记录结果、平台契约版本和重试建议。

逐记录结果至少包含 dataset、record_key、idempotency_key、payload_hash、结果、平台实体类型/ID、错误码、字段路径、消息、retryable、review_task_id 和 applied_at。

平台还需回传：

- 数据侧 record key 与平台业务 ID 映射；
- 商品/公司匹配候选的 accepted/rejected/deferred 决定；
- 正式分类决定和平台字段修订要求；
- 错误记录重试范围及 retry_after。

## 17. 平台数据库接入原则

推荐顺序：内部导入 API → 权限隔离的 `ingest/staging` → L3 文件导入。

若使用 staging：

- 数据侧账号只能写 staging/ingest，不拥有正式业务表写权限；
- staging 原样保存 delivery ID、record key、幂等键、payload hash 和 lineage；
- 平台校验作业负责转换、审核、写正式表和生成 receipt；
- 数据侧不得根据历史 SQL 快照猜测或直接执行正式迁移；
- 正式业务 ID 由平台生成并通过 receipt 回传；
- 正式表不强制采用本文逻辑数据集名称。

当前平台源码、正式 DDL 和 `database/public.sql` 均未在工作区提供，因此正式表、真实唯一键、长度、枚举和索引必须在平台仓库到达后共同定稿。

## 18. 全量覆盖设计

“全量”定义为：在确认的来源范围、平台类目、关键词、店铺扩展和公开/授权边界内，所有分区完成遍历并有覆盖台账，而不是声称抓取整个平台所有商品。

首次基线：

```text
类目树快照
→ 平台专用关键词计划
→ 类目/关键词/店铺分区
→ 列表全遍历
→ 商品详情和 SKU
→ 店铺
→ 公司/厂家/资质
→ 媒体
→ L0-L2 质量核对
→ accepted baseline
```

覆盖台账至少记录 source、scope、类目、关键词、店铺、分区、分页/游标、预计/实际数量、唯一数、有效率、检查点、首次/最近成功时间和停止原因。

只有完整遍历或来源明确状态才能确认下架；受限、解析失败、局部扫描和关键词漏覆盖不能产生下架。

## 19. 分层更新与效率

| 数据 | 推荐更新方式 |
|---|---|
| 商品存在/上下架、列表摘要 | 高频轻量分区扫描 |
| 价格、库存、销量、供货状态 | 按来源允许频率保存快照 |
| 标题、主图、SKU 摘要 | 内容哈希变化后补抓详情 |
| 完整详情与参数 | 中低频或变化触发 |
| 店铺资料 | 低频，变更触发补采 |
| 公司和厂家事实 | 独立低频更新 |
| 工商/资质 | 更低频、到期前或来源变化时更新 |
| 类目树、字段集合、枚举 | 定期健康巡检并版本化 |
| 长期无变化商品 | 自动降频，但定期全量重扫 |

效率措施：

- 先用列表摘要、来源更新时间或内容哈希做低成本变化判断；
- 图片按内容哈希跨批次去重，仍保留每个来源和角色关系；
- 按类目、关键词、页、商品 ID、店铺 ID 和公司 ID 保存检查点；
- 失败只补失败分区；
- 商品、资质和媒体队列分别设置并发与速率；
- 同一来源写运行互斥，跨来源并发受 runner 资源锁限制；
- accepted baseline 和增量水位只在质量通过并 `completed` 后推进。

## 20. 质量门禁

质量规则分为共享规则和来源能力规则。

共享规则：

- 记录数量、成功数、唯一数、重复率；
- 来源 ID、来源 URL、采集时间和证据覆盖；
- L0/L1/L2 数量可核对；
- 图片/媒体成功、失败和占位状态；
- 新增、修改、疑似缺失、明确下架和无变化；
- Schema、字段集合、枚举和页面结构漂移；
- run_result、manifest、checksums 和质量报告完整。

来源能力规则：

- 1688 可要求公司/资质路径覆盖；
- 淘宝不能因缺少 1688 专有厂家字段而失败，但历史稳定字段突然归零必须失败；
- 京东资质遇到滑块应进入人工门禁，不计为成功空值；
- 闲鱼弱类目应提高范围复核比例；
- 拼多多标题/属性二次判定必须记录污染和排除率。

数量为 0、异常大幅下降、关键字段大面积缺失、价格批量归零、解析漂移、下架异常或质量报告不完整时，禁止交付并禁止推进 baseline。

## 21. 测试和晋级

### 21.1 离线门禁

- dry-run 零网络、零浏览器、零运行资产副作用；
- fixture replay 可重复，除约定时间字段外哈希一致；
- 正常状态、全部异常状态和 Schema 故障路由均有测试；
- 重复触发、锁冲突、过期锁和检查点恢复有测试；
- 所有 adapter 通过统一 contract tests。

### 21.2 在线小样本

- 人工授权、低频、可观察；
- 验证正常、登录失效、验证码/滑块、429、临时 5xx、parser drift 和人工恢复；
- 不扩大未通过的路径。

### 21.3 分区扩量

单关键词/类目 → 多分区 → 全量候选。每一级都核验锁、断点、数量、质量、证据和重放。

### 21.4 启用和 30 天验收

一次只启用一个来源进入观察期。30 天至少证明：

- 重复触发不产生重复运行；同来源写运行无重叠；
- 所有计划任务都有完成、人工等待或明确失败结果；
- 崩溃、5xx、登录失效、验证码、429、parser drift 和质量拒绝均完成演练；
- 失败不推进 baseline、水位或下架；
- full/增量可由 L0 重放并复现 L1-L3；
- 告警可去重、升级并发送恢复通知；
- 零凭据泄露、零风控绕过、零正式业务表直写；
- 至少一次通过的 full 和多次通过的增量/分区复采。

## 22. 来源建设顺序

1. 先确认 n8n 部署、Windows/browser runner、文件挂载和原子锁存储。
2. 收紧 `run_result` 并建立 request/stage/checkpoint/review/L3 schemas 与 contract tests。
3. 使用 fixtures 建设完全离线的 shared 控制面。
4. 漫立方先 replay 现有正式批次，验证 L0-L3 重建和质量/回执；授权 API 到达后再加入事件增量。
5. 1688 作为丰富商品—店铺—公司—资质参考，先小样本、再多商品/多公司、再分区全量。
6. 淘宝补齐 L0/L2、检查点、停止语义和统一结果，验证稀疏厂家字段。
7. 京东验证商品自动链路与资质人工门禁。
8. 拼多多、抖音、闲鱼分别验证扁平类目、动态类目和弱类目/二手场景。
9. 平台合同确认后接入 L3 API/staging/receipt。
10. 单来源逐个进入 30 天稳定性验收。

## 23. 当前契约漂移和实施阻断项

| 问题 | 当前事实 | 影响 | 关闭要求 |
|---|---|---|---|
| n8n workflow | 只有 README，无 JSON | 不能启用来源 | shared 离线控制面和来源 JSON 经过审查 |
| `run_result` | Schema 宽松，1688 输出字段不一致 | n8n 无法安全路由 | 严格 Schema、适配器统一输出、contract tests |
| 漫立方入口 | 无统一 run_source/dry-run/run_result | 不能接 shared runner | 先做离线 replay 入口 |
| 淘宝资产层 | 当前主要是 L1 CSV | 无可靠恢复、变化和质量 | 补齐 L0、L2、检查点和停止语义 |
| 淘宝受限处理 | 受限后可能继续遍历 | 可能误判空值/下架 | 首次受限停止并映射具体状态 |
| 1688 面积字段 | “工厂面积”“厂房面积”共用 `factory_area_sqm` | 语义污染 | 新增 `factory_building_area_sqm` 并更新代码、Schema、测试 |
| 漫立方 v2 投影 | builder 表头/分类与当前 v2 文档存在漂移 | 不能宣称当前交付是 v2 样板 | 对照真实平台契约重做 L3 adapter 或修正规范版本 |
| 正式数据库 | 平台源码和 DDL 未提供，database dump 不存在 | 无法定稿正式表和索引 | 平台同事提供仓库/契约并确认 P0 项 |
| 部署拓扑 | n8n 与 Windows 浏览器 runner 未定 | 无法安全调用 profile | 先确定单机开发和长期运行拓扑 |

这些阻断项进入后续唯一建设路线图，不在多个临时计划中重复维护。

## 24. 平台接入前 P0 确认清单

用户和平台负责人必须共同确认：

1. 来源商品刊登、游艺圈产品、配件和聚合产品的准确语义。
2. 哪些实体允许自动进入待审核区，哪些必须人工确认。
3. 平台业务 ID 生成方、数据侧 UID 接受方式和 ID 回传。
4. 每个平台稳定键、键版本、ID 变化和别名迁移。
5. 批次与记录幂等键的作用域和保留期限。
6. 首期使用内部 API、权限隔离 staging 还是文件导入。
7. contract、实体 Schema、分类和属性字典版本兼容规则。
8. 必填、长度、decimal 精度、币种、单位和时区。
9. present/missing/invalid/conflict/redacted 及缺失原因枚举。
10. 主分类、适配机型、形态、玩法和状态枚举。
11. 店铺—公司—厂家/供应商/品牌方关系枚举。
12. 工商主体、资质核验状态和可公开展示条件。
13. full/delta、疑似缺失、明确下架、重上架、撤回和回滚语义。
14. 图片/文档对象存储、权限、版权、尺寸和校验和。
15. 质量阻断阈值和匹配/AI 置信度阈值。
16. 复核任务归属、状态、SLA 和结果回传。
17. 批次/记录回执、错误码、重试和平台 ID mapping。
18. 联系方式、营业执照等敏感数据的接收权限和脱敏。

这些确认完成前，数据侧可以建设 L0-L2 和 L3 候选契约，但不能把推荐逻辑模型当作正式数据库定稿。
