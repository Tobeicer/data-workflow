# 游艺圈数据工作流总纲

状态：文档合并设计已确认，待执行内容迁移

日期：2026-07-15

## 0. 文档收敛设计

### 0.1 目标

将当前分散在执行基线、总体技术设计、建设路线图和 AI 交接文件中的现行内容，合并为本文件这一份权威总纲。合并后，项目成员和 AI 只需读取总纲即可理解项目边界、技术设计、实施顺序、当前状态和下一执行点。

### 0.2 最终现行文档结构

合并完成后只保留以下现行入口：

| 文件 | 唯一职责 |
|---|---|
| `README.md` | 面向人员的极简项目入口，只链接总纲与专项参考 |
| `AGENTS.md` | 面向 AI 的工作区规则、保护边界和最小阅读顺序 |
| `docs/游艺圈数据工作流总纲.md` | 唯一现行总纲：项目边界、技术设计、实施路线图、状态和下一执行点 |
| `docs/游艺圈游戏游艺设备完整分类清单.md` | 分类、平台类目映射、关键词和范围规则专项参考 |
| `游艺圈数据导入字段规范_v2.md` | 当前 L3 Excel 兼容契约 |
| `data-workflow/adapters/<source>/README.md` | 各来源特有边界、能力、命令和运行证据 |
| `data-workflow/orchestration/n8n/configs/source_registry.json` | 七来源状态和启用值的机器权威 |

`docs/project-split/` 与 `docs/requirements/` 继续作为受保护历史资料存在，但不属于现行执行入口；只有 `docs/requirements/信息整理.md` 接收新的已确认业务需求。

### 0.3 总纲内部结构

最终总纲按以下固定顺序组织：

1. 文档权威、阅读方式和项目定位。
2. 角色职责、数据边界和 L0-L3 分层。
3. 共享控制面、独立适配器和运行状态机。
4. 公共核心字段、类型化属性、原始全字段和逻辑数据集。
5. 分类、质量、安全、全量/增量和平台接入规则。
6. 实施路线图、任务依赖、验收和提交要求。
7. 当前状态、下一执行点和完成定义。

总纲只保留一次权威定义。原三份文档中重复的权威顺序、项目定位、职责边界、L0-L3、数据库禁止写入、来源定位、质量门禁和文档治理规则合并去重；技术细节和未完成任务不得因去重丢失。

### 0.6 合并规则

- 以用户最新指令为最高优先级。
- 已确认技术语义原样保留，必要时只调整标题层级、交叉引用和重复表达。
- 建设路线图的任务编号、依赖、文件路径、测试命令、验收条件和当前勾选状态全部保留。
- 当前执行点保持为 A1：核验 n8n、执行器、运行目录和锁存储部署拓扑。
- 数据侧逻辑数据集继续明确为建议契约，不转换成正式数据库 DDL 或写库授权。
- 七来源保持 `enabled=false`，直到逐来源启用门禁通过并得到明确批准。
- 不把命令失败、对话过程、已否决方案或迁移过程日志写入最终总纲。
- 总纲目录使用稳定锚点，其他入口只链接对应章节，不复制内容。

### 0.7 验收条件

合并只有同时满足以下条件才算完成：

1. 旧三份文档和 `AI_HANDOFF.md` 已删除。
2. 总纲包含原执行基线的全部稳定决策、原技术设计的全部逻辑模型与控制面设计、原路线图的全部未完成任务。
3. 现行 Markdown 与测试不再引用旧文件路径。
4. `README.md` 和 `AGENTS.md` 的默认阅读路径只包含总纲与按需专项参考。
5. `docs/project-split/` 和受保护 requirements 未发生修改。
6. 七来源 registry 状态和 adapter 当前事实没有被文档合并改变。
7. 无未决占位语句、无敏感值、无伪启用状态。
8. `git diff --check` 和完整 pytest 通过。

### 0.8 实施方法

先用脚本化、可复核的标题降级和内容拼接形成总纲草稿，再人工去除重复章节并修正交叉引用；随后更新所有入口和治理测试，删除旧文件，最后执行全文检索、保护目录检查和完整测试。合并过程不修改爬虫代码、机器契约、来源状态或运行资产。
