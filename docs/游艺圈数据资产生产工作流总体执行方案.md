# 游艺圈数据资产生产工作流总体执行方案

版本：V1.2  
日期：2026-07-13  
定位：数据生产线的运行架构与长期验收依据

职责、L0-L3 和正式写库边界以 `数据工作流与游艺圈系统对接执行基线.md` 为准。本文不重复定义职责，只说明如何稳定运行。

## 1. 总体架构

```text
定时 / Webhook / 人工 / 事件
                ↓
           n8n 控制面
                ↓
     来源适配器与通用脚本
                ↓
        L0 → L1 → L2
                ↓
       质量门禁 / 人工门禁
                ↓
              L3
                ↓
       平台导入、审核与发布
```

n8n 负责触发、依赖、锁、超时、重试、状态、人工确认和通知。Python/Node 脚本负责采集、浏览器自动化、清洗、图片、文档、AI、增量比较、质量计算和交付包生成。

## 2. 标准状态与脚本契约

每次任务使用唯一 `run_id`：

```text
<source>_<mode>_<YYYYMMDD_HHMMSS>
```

正常状态：

```text
created → prechecking → ready → collecting → raw_archived
→ normalizing → matching → enriching → validating
→ waiting_review / packaging → delivered → completed
```

异常状态：

```text
login_required
human_verification_required
rate_limited
source_changed
parser_drift
quality_rejected
retryable_failed
manual_stopped
permanently_disabled
```

统一命令目标：

```powershell
python run_source.py precheck --config <config> --run-id <run_id>
python run_source.py collect --config <config> --run-id <run_id>
python run_source.py normalize --run-id <run_id>
python run_source.py diff --run-id <run_id> --baseline <baseline_run_id>
python run_source.py enrich --run-id <run_id>
python run_source.py validate --run-id <run_id>
python run_source.py package --run-id <run_id>
python run_source.py resume --run-id <run_id>
```

正式来源至少实现 `precheck`、`collect/import`、`validate` 和 `package`。每个阶段输出退出码与机器可读状态；n8n 不解析自然语言日志。

## 3. 触发方式

| 触发 | 适用场景 | 约束 |
|---|---|---|
| Webhook/事件 | 合作方文件、厂家上传、平台反馈 | 必须有明确授权入口 |
| 定期 | 官网、政策、文审、价格库存、来源健康 | 不用高频轮询模拟实时 |
| 人工任务 | 关键词、厂家、产品、异常补采 | 记录任务人、范围和停止条件 |
| 人工授权 | 淘宝、1688 等登录来源 | 登录失效即暂停，不绕验证码 |
| API 按需 | 企业核验、付费数据 | 缓存、配额、有效期和成本上限 |

## 4. 目录与工作流资产

目标目录以 `../data-workflow/README.md` 为准。`orchestration/` 只保存 n8n 控制面：

```text
data-workflow/orchestration/
├─ workflows/
├─ configs/
├─ schemas/
├─ prompts/
├─ fixtures/
└─ deployment/
```

可部署资产包括：主/子工作流 JSON、适配器脚本、配置、Schema、提示词、测试样本、环境变量示例和恢复说明。不得把密钥、Cookie、大数据或业务代码塞进 n8n JSON。

## 5. 来源运行矩阵

| 来源 | 用途 | 推荐模式 | 关键门禁 |
|---|---|---|---|
| 漫立方 | 配件、SKU、价格库存、图片和分类 | 周期增量、人工全量 | 原始响应、哈希、变化报告 |
| 合作厂家/供应商 | 正式厂家和产品主数据 | 文件、Webhook、API | 文件版本、字段校验、人工确认 |
| 厂家官网 | 厂家资料和产品 | 月度或专项 | 页面哈希、结构变化、robots/规则 |
| 政府/文审网站 | 政策、文审和风险信息 | 日/周监控 | 官方原文、日期、地区、复核 |
| 1688 | 新品、参数、SKU、批发价和公司线索 | 人工登录、低频专项 | 登录门禁、限频、断点、候选用途 |
| 淘宝 | 参数、价格和同款候选 | 人工授权、低频专项 | 验证码即停、结果不覆盖主来源 |
| 其他大平台 | 新品、趋势和商机线索 | 先小样本验证 | 明确可用模式和停用规则 |
| 企业信息 API | 工商主体核验 | 新候选按需 | 缓存、去重、配额、成本 |
| PDF/画册/资料 | 产品和厂家资料 | 文件到达 | 哈希、OCR/AI 草稿、人工复核 |

## 6. AI 使用规则

AI 只处理确定性规则难以稳定解决的任务：非结构化抽取、分类建议、相似候选、图片描述、缺失和复核原因。

无人值守输出必须包含：任务版本、来源标识、结构化结果、置信度、理由、缺失、冲突和复核标记。

以下结论必须人工确认：工商主体和认证、文审/法律结论、侵权判断、低置信度实体合并、高风险联系方式和可能导致下架或信用处罚的结论。

模型、提示词和 Schema 需要版本号；相同输入使用哈希缓存；设置任务和周期成本上限。

## 7. 稳定性与安全

- 同一 `run_id` 和阶段重复执行应幂等。
- 同一来源全量任务默认不并发。
- 网络超时和临时 5xx 可指数退避重试。
- 登录失效、验证码、权限不足和字段变化不盲目重试。
- 长任务按分类、分页、关键词或商品 ID 保存检查点。
- 连续 403/429、验证码或高失败率时立即熔断，保存现场并转人工或停用。
- 密钥、Cookie、数据库密码和登录态只进入安全凭据或未跟踪环境文件。
- 开发、测试、常驻环境使用不同配置和凭据。

## 8. 质量门禁

通用指标：

- 记录数量和唯一 ID 完整率；
- 重复率、必填字段和来源字段完整率；
- 图片成功率、分类覆盖率和原始/清洗可核对性；
- 新增、修改、下架、失败、冲突和复核数量。

以下情况不得自动正常交付：

- 数量异常下降或返回 200 但解析为 0；
- 必填完整率、图片成功率或分类覆盖率显著下降；
- 字段集合、页面结构或枚举变化；
- 大量价格归零、异常下架或 AI Schema 失败；
- 付费 API 或 AI 成本超预算。

## 9. 监控与报告

监控任务耗时、阶段数量、来源健康、登录有效期、配额、成本、字段变化、磁盘和最近成功时间。

告警分为：

- 信息：完成或无变化；
- 警告：少量失败、完整率下降、登录即将过期；
- 高：验证码、结构变化、解析为 0、质量门禁失败；
- 紧急：凭据泄露、异常高频请求或重大合规风险。

每次运行至少产生：

- `run_result.json`；
- `run_manifest.json`；
- 质量报告；
- 变化摘要；
- 复核队列；
- `errors.jsonl` 和必要调试证据。

## 10. 标准交付

```text
delivery_<source>_<batch>/
├─ README.md
├─ manifest.json
├─ schema.json 或字段字典.xlsx
├─ normalized_records.*
├─ relations/
├─ snapshots/
├─ images/
├─ documents/
├─ change_summary.*
├─ review_queue.*
├─ quality_report.*
└─ checksums.sha256
```

交付说明必须包含来源、批次、方式、时间、唯一键、字段、变化数量、质量、复核、文件引用、已知限制和更新频率。

## 11. 建设顺序

1. 统一目录、批次、状态和交付契约。
2. 建立 n8n 基础编排。
3. 将漫立方接入增量、无变化和失败恢复。
4. 完成 1688 公司资产试采器并接入统一工作流。
5. 接入淘宝、合作商文件、厂家资料、文审和企业核验。
6. 逐个平台验证其他补充来源。
7. 连续运行不少于 30 天，验证告警、恢复、重放和可复现交付。

达到上述状态后，基础体系进入持续运营和扩展阶段。
